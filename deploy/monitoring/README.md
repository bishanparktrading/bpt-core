# Monitoring — Prometheus + Alertmanager + ntfy.sh

Local observability stack that runs alongside the bpt-* services on the same
host. Prometheus scrapes each service's `/metrics` endpoint, evaluates alert
rules against the time series, and fires to Alertmanager. Alertmanager
groups/dedups/routes alerts and POSTs them to an ntfy.sh topic, which
pushes notifications to the operator's phone via FCM/APNs.

This is the MVP alerting stack — appropriate for dev and testnet. **Before
switching any meaningful capital to mainnet, upgrade the receiver from
ntfy.sh to PagerDuty (free tier is up to 5 users, unlimited incidents,
phone-ringing)** — the swap is a 5-line change in `alertmanager.yml`.
Tracked in the prod-hardening backlog.

## Layout

```
deploy/monitoring/
├── README.md                         (this file)
├── prometheus/
│   ├── prometheus.yml                # scrape_configs, rule_files, alertmanager
│   └── rules/
│       └── bpt-alerts.yml            # PromQL alert rules (critical + warning)
└── alertmanager/
    └── alertmanager.yml              # receivers (Discord webhook), routes, inhibit rules
```

Systemd user units (generated by `deploy/generate-units.sh`):
- `bpt-prometheus.service` — runs `prometheus --config.file=...`
- `bpt-alertmanager.service` — runs `alertmanager --config.file=...`
- `bpt-heartbeat.service` + `.timer` — curls Healthchecks.io every 5 min (dead-man's-switch for host-down case that local Alertmanager can't catch)

## Install the binaries

```
sudo apt install prometheus prometheus-alertmanager
```

Both are pure Go static binaries. Ubuntu/Debian namespace the Alertmanager
binary as `/usr/bin/prometheus-alertmanager` (avoids conflict with anything
else called "alertmanager") — that's the path the generated unit file
points at. If you installed via the upstream tarball instead, edit
`bpt-alertmanager.service` ExecStart to `/usr/local/bin/alertmanager`.

Disable the system-managed units so they don't compete with the user units:

```
sudo systemctl disable --now prometheus prometheus-alertmanager
```

Note: on Debian/Ubuntu the system unit is called `prometheus-alertmanager.service`,
not `alertmanager.service`.

## ntfy.sh topic

ntfy is HTTP pub/sub. The topic name is the secret — anyone who knows
it can publish to OR subscribe to the topic. Pick a non-guessable one:

```
TOPIC="bpt-alerts-$(uuidgen | tr -d -)"
echo "$TOPIC"   # save this — you'll paste it into the phone app + systemd-creds
```

Install the **ntfy** app on your phone (Android / iOS — both free and
open-source), then subscribe to `$TOPIC`. Verify:

```
curl -d "test from laptop" "https://ntfy.sh/$TOPIC"
```

Your phone should ping within 1-2 seconds. If it doesn't, double-check
the topic string and that the app shows a green "connected" indicator.

Then store the two severity-tiered URLs as plain files with mode 0600.
Each URL embeds priority + tag query params so the phone's notification
behaviour differs per severity (urgent bypasses Do Not Disturb if you
grant ntfy that permission):

```
mkdir -p ~/.config/bpt && chmod 0700 ~/.config/bpt

echo -n "https://ntfy.sh/$TOPIC?priority=urgent&tags=rotating_light&title=BPT+CRITICAL" > ~/.config/bpt/ntfy-url-critical
echo -n "https://ntfy.sh/$TOPIC?priority=default&tags=warning&title=BPT+WARNING"        > ~/.config/bpt/ntfy-url-warning
chmod 0600 ~/.config/bpt/ntfy-url-critical ~/.config/bpt/ntfy-url-warning
```

The generated `bpt-alertmanager.service` unit uses
`LoadCredential=bpt-ntfy-url-critical:%h/.config/bpt/ntfy-url-critical`
to copy each into `$CREDENTIALS_DIRECTORY` (/run/credentials/...) at
service start. `alertmanager.yml` references those runtime paths via
`webhook_configs.url_file`.

The source files are unencrypted at rest. This is acceptable for a
single-user laptop — systemd-creds encrypt with host key only adds
meaningful protection when the attacker can read your home dir but
not the system key, which isn't the threat model on your own laptop.
Tighten to `LoadCredentialEncrypted=` (systemd 252+ with `--user`)
when moving to a prod host.

## Dead-man's-switch

Alertmanager can't alert you when the host itself dies — it dies with it.
`bpt-heartbeat.timer` runs every 5 min and `curl`s a Healthchecks.io URL.
If Healthchecks.io doesn't hear a check-in for 10 min, **they** email /
Discord-ping you. Create a free check at https://healthchecks.io and store
the URL in `/etc/bpt/healthchecks.env`:

```
# /etc/bpt/healthchecks.env
HC_URL=https://hc-ping.com/abc123-def456-...
```

## Bring-up

```
sudo apt install prometheus prometheus-alertmanager
./deploy/generate-units.sh          # emits bpt-prometheus/alertmanager/heartbeat units
systemctl --user daemon-reload
systemctl --user enable --now bpt-prometheus.service bpt-alertmanager.service bpt-heartbeat.timer
```

## Verify

- Prometheus UI: http://localhost:9090 — check `Status → Targets` (all should be UP)
- Alertmanager UI: http://localhost:9093 — shows active + silenced alerts
- Trip a test alert: temporarily lower a threshold in `bpt-alerts.yml`, rerun
  `systemctl --user reload bpt-prometheus`, watch it appear in Alertmanager
  and fire through to Discord.

## Bestiary of alert rules

See `prometheus/rules/bpt-alerts.yml` — each rule carries a `severity` label
(critical | warning), a free-text `summary`, and a `description` templated
with the metric value. Alertmanager routes critical to Discord
immediately (no grouping delay); warnings are batched 2-minute windows to
reduce noise during minor blips.

## When this setup outgrows local-host

This setup is Option A from the design conversation — same host as services.
The upgrade path to Option B (separate observability host) is just moving
`prometheus.yml` / `alertmanager.yml` / `bpt-alerts.yml` to a VPS and
pointing `scrape_configs` at the trading host over Tailscale. No config
rewrite, no rules change.
